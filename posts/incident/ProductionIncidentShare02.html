生产事故分享02：线程池自定义Thread finally块不被调用问题回溯

📋 事故概览

项目详情
故障时间2025年X月X日 XX:XX - XX:XX
影响时长约30分钟
故障等级P2（中等）
影响范围上下文清理功能失效
故障现象自定义SystemThread的finally块未按预期每次任务后执行，导致上下文数据未及时清理

🎯 需求背景

业务系统使用线程池处理异步任务，为了在每次任务执行后进行上下文清理，开发团队：

  • 自定义了SystemThreadFactory创建特殊线程
  • 继承Thread实现SystemThread，重写run()方法
  • 期望在SystemThread.run()的finally块中进行上下文清理

🔍 问题排查过程

现象描述

​ 性能团队在灰度环境进行了一轮压测，然后产品运营开始在灰度环境进行需求验收，但是在验收过程中发现访问一个页面会偶现访问到压测数据库的数据。

详细排查步骤

因为我们系统实际连了两个数据库，一个是正常业务数据库，还有一个压测数据库，当rpc请求的线程上下文有某个压测标识的参数时，此时在DAO层会进行AOP连接，获取到压测数据库的连接进行压测。

但是在这个问题发生的时候，运营是在正常的灰度环境，没有走压测请求流程，我们第一时间反应应该是系统里面某个线程上下文清理有问题，但是压测请求带进来的上下文没有清理干净。

然后就开始针对请求处理流程进行代码分析，结果 发现系统在处理一个业务逻辑时使用了线程池，而这个线程池自定义了线程工厂和线程。问题代码如下

// 自定义线程工厂
public class SystemThreadFactory implements ThreadFactory {
    @Override
    public Thread newThread(Runnable r) {
        return new SystemThread(r, "system-thread-" + threadNumber.getAndIncrement());
    }
}

// 自定义线程类 - 问题所在！
public class SystemThread extends Thread {
    public SystemThread(Runnable target, String name) {
        super(target, name);
    }
    
    @Override
    public void run() {
        try {
            super.run(); // 这里只会执行一次！
        } finally {
            // 期望每个任务后执行，实际只在线程退出时执行
            Context.removeContext();
        }
    }
}

刚开始看到这个代码没有意识到问题所在，认为每次任务执行，SystemThread的run方法finally块都会执行，但是已经排除了其他地方的可能性，只能开始写Demo验证finally块是否会执行，结果，debug发现finally块永远都不会执行，除非主动关闭线程池。然后就开始分析背后的原因......

🐛 问题根因分析

线程池执行流程分析

Worker类核心源码

private final class Worker extends AbstractQueuedSynchronizer implements Runnable {
    final Thread thread;
    Runnable firstTask;
    
    Worker(Runnable firstTask) {
        this.firstTask = firstTask;
        // 关键：传入的是Worker本身，不是业务任务！
        this.thread = getThreadFactory().newThread(this);
    }
    
    public void run() {
        runWorker(this); // Worker的run方法
    }
}

runWorker方法核心逻辑

final void runWorker(Worker w) {
    Thread wt = Thread.currentThread();
    Runnable task = w.firstTask;
    
    try {
        // 关键循环：在同一个线程中执行多个任务
        // 线程池中的线程会一直按生产者消费者模型处理提交的任务
        while (task != null || (task = getTask()) != null) {
            w.lock();
            try {
                beforeExecute(wt, task);
                task.run(); // 直接调用任务的run方法
                afterExecute(task, null);
            } finally {
                task = null;
                w.unlock();
            }
        }
    } finally {
        processWorkerExit(w, completedAbruptly);
    }
}

根因总结

  1. 线程生命周期误解：线程池中的线程只会执行一次run()方法，而不是每个任务都调用
  2. Worker机制不理解：线程池通过Worker的runWorker循环执行多个任务，任务直接调用task.run()
  3. finally执行时机错误：SystemThread.run()的finally只在线程退出时执行，无法实现每任务清理

💡 解决方案

临时解决方案

立即让运维重启服务，清理线程池中线程的上下文，并通知性能团队，在新版本上线前不要进行压测。

永久解决方案

方案1：任务装饰器模式（推荐）

public class ContextCleanupRunnable implements Runnable {
    private final Runnable delegate;
    
    public ContextCleanupRunnable(Runnable delegate) {
        this.delegate = delegate;
    }
    
    @Override
    public void run() {
        try {
            delegate.run();
        } finally {
            // 每个任务执行后清理上下文
            Context.removeContext();
            log.debug("Context cleaned for task: {}", delegate.getClass().getSimpleName());
        }
    }
}

// 使用工具类简化
public class TaskWrapper {
    public static Runnable withContextCleanup(Runnable task) {
        return new ContextCleanupRunnable(task);
    }
}

方案2：自定义ThreadPoolExecutor

public class ContextAwareThreadPoolExecutor extends ThreadPoolExecutor {
    
    public ContextAwareThreadPoolExecutor(int corePoolSize, int maximumPoolSize,
                                         long keepAliveTime, TimeUnit unit,
                                         BlockingQueue<Runnable> workQueue) {
        super(corePoolSize, maximumPoolSize, keepAliveTime, unit, workQueue);
    }
    
    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        super.afterExecute(r, t);
        // 每个任务执行后清理上下文
        Context.removeContext();
        log.debug("Context cleaned after task execution");
    }
}

方案3：使用ThreadLocal清理钩子

public class ContextManager {
    private static final ThreadLocal<Set<String>> CONTEXT_KEYS = new ThreadLocal<>();
    
    public static void setContext(String key, Object value) {
        // 记录设置的key
        Set<String> keys = CONTEXT_KEYS.get();
        if (keys == null) {
            keys = new HashSet<>();
            CONTEXT_KEYS.set(keys);
        }
        keys.add(key);
        
        // 设置实际值
        ThreadLocalContext.set(key, value);
    }
    
    public static void clearAllContext() {
        Set<String> keys = CONTEXT_KEYS.get();
        if (keys != null) {
            keys.forEach(ThreadLocalContext::remove);
            CONTEXT_KEYS.remove();
        }
    }
}

📚 经验总结

🎯 核心教训

  1. 线程池工作原理深度理解

    • 线程池中的线程是复用的，不是每个任务创建新线程
    • Worker机制通过循环在同一线程中执行多个任务
    • 自定义Thread的run()方法只在线程启动时执行一次

  2. ThreadFactory使用场景

    • ThreadFactory主要用于设置线程名称、优先级、守护线程等属性
    • 不适合用于任务级别的逻辑处理
    • 线程级别的资源管理与任务级别的资源管理要区分清楚

  3. 上下文管理最佳实践

    • 任务级别的资源清理应该在任务执行层面处理
    • 使用装饰器模式或ThreadPoolExecutor的钩子方法
    • ThreadLocal的清理时机要准确把握

🔧 最佳实践代码模板

@Component
public class AsyncTaskManager {
    
    private final ThreadPoolExecutor executor;
    
    public AsyncTaskManager() {
        this.executor = new ContextAwareThreadPoolExecutor(
            10, 20, 60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(1000),
            new ThreadFactoryBuilder()
                .setNameFormat("async-task-%d")
                .setDaemon(false)
                .build()
        );
    }
    
    public void submitTask(Runnable task) {
        executor.submit(TaskWrapper.withContextCleanup(task));
    }
    
    public <T> Future<T> submitTask(Callable<T> task) {
        return executor.submit(TaskWrapper.withContextCleanup(task));
    }
    
    @PreDestroy
    public void shutdown() {
        executor.shutdown();
        try {
            if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
                executor.shutdownNow();
            }
        } catch (InterruptedException e) {
            executor.shutdownNow();
            Thread.currentThread().interrupt();
        }
    }
}

// 通用任务包装工具
public class TaskWrapper {
    
    public static Runnable withContextCleanup(Runnable task) {
        return () -> {
            try {
                task.run();
            } finally {
                ContextManager.clearAllContext();
            }
        };
    }
    
    public static <T> Callable<T> withContextCleanup(Callable<T> task) {
        return () -> {
            try {
                return task.call();
            } finally {
                ContextManager.clearAllContext();
            }
        };
    }
}
