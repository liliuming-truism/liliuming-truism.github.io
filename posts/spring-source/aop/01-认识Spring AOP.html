AOP 的诞生背景

横切关注点问题

// 1990年代典型的代码风格
public class OrderService {
    
    public void createOrder(Order order) {
        // 日志记录 - 横切关注点1
        System.out.println("开始创建订单: " + order.getId());
        
        // 权限检查 - 横切关注点2
        if (!SecurityContext.hasPermission("ORDER_CREATE")) {
            throw new SecurityException("无权限");
        }
        
        // 事务开始 - 横切关注点3
        Connection conn = null;
        try {
            conn = dataSource.getConnection();
            conn.setAutoCommit(false);
            
            // 性能监控 - 横切关注点4
            long startTime = System.currentTimeMillis();
            
            // ========== 核心业务逻辑（只有这2行）==========
            orderDao.insert(order);
            inventoryService.deduct(order.getItems());
            // =============================================
            
            // 性能监控
            long endTime = System.currentTimeMillis();
            System.out.println("执行耗时: " + (endTime - startTime) + "ms");
            
            // 事务提交
            conn.commit();
            
            // 日志记录
            System.out.println("订单创建成功: " + order.getId());
            
        } catch (Exception e) {
            // 异常处理 - 横切关注点5
            if (conn != null) {
                conn.rollback();
            }
            System.err.println("订单创建失败: " + e.getMessage());
            throw e;
        } finally {
            // 资源释放
            if (conn != null) {
                conn.close();
            }
        }
    }
    
    public void updateOrder(Order order) {
        // 又要重复一遍所有的横切逻辑...
        System.out.println("开始更新订单: " + order.getId());
        if (!SecurityContext.hasPermission("ORDER_UPDATE")) {
            throw new SecurityException("无权限");
        }
        // ... 重复的代码
    }
    
    public void deleteOrder(Long orderId) {
        // 再重复一遍...
        System.out.println("开始删除订单: " + orderId);
        if (!SecurityContext.hasPermission("ORDER_DELETE")) {
            throw new SecurityException("无权限");
        }
        // ... 重复的代码
    }
}

// 其他服务也要重复这些逻辑
public class UserService {
    public void createUser(User user) {
        // 又是一样的日志、权限、事务、监控代码...
        System.out.println("开始创建用户: " + user.getId());
        if (!SecurityContext.hasPermission("USER_CREATE")) {
            throw new SecurityException("无权限");
        }
        // ...
    }
}

以上代码存在如下问题：

  1. 代码重复率高达 80%

  2. 核心业务逻辑被淹没

  3. 修改横切逻辑需要改动所有文件

  4. 难以维护和测试

  5. 违反 DRY 原则（Don't Repeat Yourself）

OOP的解决方案及其局限性

使用继承

public abstract class BaseService {
    protected void logBefore(String method) {
        System.out.println("开始执行: " + method);
    }
    protected void logAfter(String method) {
        System.out.println("执行完成: " + method);
    }
}

public class OrderService extends BaseService {
    public void createOrder(Order order) {
        logBefore("createOrder");  // 😢 还是要手动调用
        // 业务逻辑
        logAfter("createOrder");   // 😢 还是要手动调用
    }
}

局限性：

  1. ❌ 仍然需要手动调用横切逻辑
  2. ❌ Java 只支持单继承，无法同时继承多个横切关注点
  3. ❌ 侵入性强，业务代码与技术代码耦合
  4. ❌ 无法对第三方类应用横切逻辑

使用装饰器模式

public interface OrderService {
    void createOrder(Order order);
}

public class OrderServiceImpl implements OrderService {
    public void createOrder(Order order) {
        // 纯粹的业务逻辑
    }
}

public class LoggingOrderService implements OrderService {
    private OrderService delegate;
    
    public void createOrder(Order order) {
        System.out.println("开始创建订单");
        delegate.createOrder(order);
        System.out.println("订单创建完成");
    }
}

public class TransactionalOrderService implements OrderService {
    private OrderService delegate;
    
    public void createOrder(Order order) {
        beginTransaction();
        delegate.createOrder(order);
        commitTransaction();
    }
}

局限性：

  1. ❌ 需要为每个服务创建多个装饰器类 → 类爆炸
  2. ❌ 装饰器的顺序需要手动管理
  3. ❌ 代码量巨大（每个服务 × 每个横切关注点）
  4. ❌ 不够灵活，难以动态配置

AOP的诞生

Gregor Kiczales 的洞察：

"传统的编程范式（OOP）在处理横切关注点时存在根本性的局限。 我们需要一种新的编程范式，能够将横切关注点模块化。"

AOP的核心思想

┌─────────────────────────────────────────────────────┐
│              AOP 的核心思想                           │
└─────────────────────────────────────────────────────┘

传统 OOP：
    纵向分解（按功能模块）
    ├─ UserService
    ├─ OrderService
    └─ ProductService
    
    问题：横切关注点散布在各个模块中

AOP：
    横向切入（按关注点）
    ├─ 日志切面     → 统一管理所有日志
    ├─ 事务切面     → 统一管理所有事务
    ├─ 安全切面     → 统一管理所有权限
    └─ 监控切面     → 统一管理所有监控
    
    优势：关注点分离（Separation of Concerns）

┌──────────────┐
│ UserService  │
├──────────────┤     ┌─────────────┐
│ OrderService │ ←───┤ 日志切面     │
├──────────────┤     ├─────────────┤
│ProductService│ ←───┤ 事务切面     │
└──────────────┘     ├─────────────┤
                     │ 安全切面     │
                     └─────────────┘
                     
横切关注点统一管理，业务代码保持纯净

AOP 解决的核心问题

1. 关注点分离（Separation of Concerns）

业务关注点              技术关注点
┌─────────────┐        ┌─────────────┐
│ 订单管理     │        │ 日志记录     │
│ 用户管理     │   ←→   │ 事务管理     │
│ 库存管理     │        │ 权限控制     │
└─────────────┘        └─────────────┘
   独立演进              独立演进

2. 代码复用

// 一个切面，应用到所有需要的地方
@Aspect
@Component
public class TransactionAspect {
    @Around("@annotation(Transactional)")
    public Object manageTransaction(ProceedingJoinPoint joinPoint) {
        // 统一的事务管理逻辑
        // 应用到所有 @Transactional 方法
    }
}

3. 动态增强

// 无需修改原有代码，动态添加新功能
@Aspect
@Component
public class CacheAspect {
    // 后期添加缓存功能，不影响原有代码
    @Around("execution(* com.example.service.*.*(..))")
    public Object cache(ProceedingJoinPoint joinPoint) {
        // 缓存逻辑
    }
}

理解 Spring AOP核心概念

我们先看一个Spring AOP的使用示例

@Aspect  // 声明这是一个切面.
@Component  // 注册为 Spring Bean
@Order(1)  // 指定优先级
public class LoggingAspect {

    private static final Logger logger = LoggerFactory.getLogger(LoggingAspect.class);

    // 定义切点：service 层的所有方法
    @Pointcut("execution(* top.truism.blog.spring.aop.*.*(..))")
    public void serviceLayer() {
    }

    // 定义切点：带有 @Log 注解的方法
    @Pointcut("@annotation(top.truism.blog.spring.aop.Log)")
    public void logAnnotation() {
    }

    // 组合切点
    @Pointcut("serviceLayer() || logAnnotation()")
    public void loggableOperations() {
    }

    // 前置通知：记录方法调用
    @Before("loggableOperations()")
    public void logBefore(JoinPoint joinPoint) {
        String className = joinPoint.getTarget().getClass().getSimpleName();
        String methodName = joinPoint.getSignature().getName();
        Object[] args = joinPoint.getArgs();

        logger.info("调用方法: {}.{}，参数: {}",
            className, methodName, Arrays.toString(args));
    }

    // 返回后通知：记录返回值
    @AfterReturning(pointcut = "loggableOperations()", returning = "result")
    public void logAfterReturning(JoinPoint joinPoint, Object result) {
        String methodName = joinPoint.getSignature().getName();
        logger.info("方法 {} 返回: {}", methodName, result);
    }

    // 异常通知：记录异常
    @AfterThrowing(pointcut = "loggableOperations()", throwing = "ex")
    public void logAfterThrowing(JoinPoint joinPoint, Exception ex) {
        String methodName = joinPoint.getSignature().getName();
        logger.error("方法 {} 异常: {}", methodName, ex.getMessage(), ex);
    }

    // 环绕通知：性能监控
    @Around("loggableOperations()")
    public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
        String methodName = joinPoint.getSignature().getName();
        long startTime = System.currentTimeMillis();

        try {
            Object result = joinPoint.proceed();
            long endTime = System.currentTimeMillis();
            logger.info("方法 {} 执行耗时: {}ms", methodName, (endTime - startTime));
            return result;
        } catch (Exception ex) {
            logger.error("方法 {} 执行失败", methodName, ex);
            throw ex;
        }
    }
}

@Component
@Slf4j
public class UserService {

    // 连接点1: createUser 方法执行

    public void createUser(String username) {
        // 连接点2: 方法执行前

在这个例子中，我们通过@Aspect声明了一个AOP切面，切面中需要包含切点 Pointcut 和 通知 Advice，下面我将详细一一介绍这些概念的含义。

Target Object

定义：被增强的原始对象，即你的业务逻辑类。

例如上面例子中方法中带有@Log注解的UserService类

理解要点：

  • 目标对象就是你原本的业务代码
  • 它不知道自己会被 AOP 增强
  • 保持业务逻辑的纯粹性

Joint Point

定义：程序执行过程中能够插入切面的点。在 Spring AOP 中，连接点总是方法的执行。

例如在上面的例子中，下面这些都是连接点

​ 方法执行前

​ 方法正常返回

​ 方法异常返回

​

理解要点：

  • Join Point 是所有可能被拦截的点
  • Spring AOP 只支持方法级别的连接点
  • AspectJ 支持更多：字段访问、构造器调用等

Pointcut

定义：从众多连接点中筛选出来的，真正需要应用通知的连接点。

类比：

  • Join Point = 所有可以安装监控的位置（100个位置）
  • Pointcut = 你决定安装监控的位置（只选了10个重要位置）

常用切点表达式：

表达式说明示例
execution匹配方法执行execution(* com.example..*.*(..))
within匹配类型within(com.example.service.*)
this匹配代理对象类型this(com.example.service.UserService)
target匹配目标对象类型target(com.example.service.UserService)
args匹配参数类型args(java.lang.String, ..)
@annotation匹配方法注解@annotation(com.example.Log)
@within匹配类注解@within(org.springframework.stereotype.Service)

execution表达式是最常用的

execution(modifiers? return-type declaring-type?method-name(params) throws?)

通配符说明：

通配符说明示例
*匹配任意字符（不含包分隔符）*.service.*
..匹配任意字符（含包分隔符）com.example..service.*
+匹配指定类及其子类com.example.BaseService+

Advice

定义：在切点处执行的具体操作，即"做什么"和"何时做"。

Before（前置通知）

执行时机：目标方法执行之前

@Aspect
@Component
public class SecurityAspect {
    
    @Before("execution(* com.example.service.*.*(..))")
    public void checkPermission(JoinPoint joinPoint) {
        String methodName = joinPoint.getSignature().getName();
        System.out.println("【前置通知】检查权限: " + methodName);
        
        // 实际应用：权限校验
        // if (!hasPermission()) {
        //     throw new SecurityException("无权限访问");
        // }
    }
}

特点：

  • 无法阻止方法执行（除非抛异常）
  • 无法修改方法参数
  • 无法获取返回值

After（后置通知）

执行时机：目标方法执行之后（无论成功还是异常）

@Aspect
@Component
public class ResourceAspect {
    
    @After("execution(* com.example.service.*.*(..))")
    public void releaseResource(JoinPoint joinPoint) {
        System.out.println("【后置通知】释放资源");
        
        // 实际应用：资源清理
        // connectionPool.release();
        // fileHandle.close();
    }
}

特点：

  • 类似 finally 块
  • 无论方法成功或异常都会执行
  • 无法获取返回值
  • 无法捕获异常

AfterReturning（返回后通知）

执行时机：目标方法正常返回之后

@Aspect
@Component
public class CacheAspect {
    
    @AfterReturning(
        pointcut = "execution(* com.example.service.UserService.getUser*(..))",
        returning = "result"  // 绑定返回值
    )
    public void cacheResult(JoinPoint joinPoint, Object result) {
        String methodName = joinPoint.getSignature().getName();
        System.out.println("【返回后通知】缓存结果: " + result);
        
        // 实际应用：缓存返回值
        // cache.put(generateKey(joinPoint), result);
    }
}

特点：

  • 只在方法正常返回时执行
  • 可以获取返回值（但不能修改）
  • 方法抛异常时不执行

AfterThrowing（异常通知）

执行时机：目标方法抛出异常之后

@Aspect
@Component
public class ExceptionAspect {
    
    @AfterThrowing(
        pointcut = "execution(* com.example.service.*.*(..))",
        throwing = "ex"  // 绑定异常对象
    )
    public void handleException(JoinPoint joinPoint, Exception ex) {
        String methodName = joinPoint.getSignature().getName();
        System.out.println("【异常通知】方法异常: " + methodName);
        System.out.println("异常信息: " + ex.getMessage());
        
        // 实际应用：异常处理
        // logger.error("方法执行异常", ex);
        // alertService.sendAlert(ex);
        // 注意：不能阻止异常传播
    }
}

特点：

  • 只在方法抛异常时执行
  • 可以获取异常对象
  • 可以指定捕获特定类型的异常
  • 不能阻止异常传播（异常仍会抛出）

指定异常类型

// 只捕获 IllegalArgumentException
@AfterThrowing(
    pointcut = "execution(* com.example.service.*.*(..))",
    throwing = "ex"
)
public void handleIllegalArgument(JoinPoint joinPoint, IllegalArgumentException ex) {
    System.out.println("捕获到参数异常: " + ex.getMessage());
}

Around（环绕通知）

执行时机：包围目标方法执行，最强大的通知类型

@Aspect
@Component
public class PerformanceAspect {
    
    @Around("execution(* com.example.service.*.*(..))")
    public Object measurePerformance(ProceedingJoinPoint joinPoint) throws Throwable {
        String methodName = joinPoint.getSignature().getName();
        
        // 前置处理
        System.out.println("【环绕通知-前】开始执行: " + methodName);
        long startTime = System.currentTimeMillis();
        
        Object result = null;
        try {
            // 执行目标方法（必须调用，否则目标方法不执行）
            result = joinPoint.proceed();
            
            // 返回后处理
            long endTime = System.currentTimeMillis();
            System.out.println("【环绕通知-后】执行成功，耗时: " + (endTime - startTime) + "ms");
            
            // 可以修改返回值
            // return modifyResult(result);
            
            return result;
            
        } catch (Exception ex) {
            // 异常处理
            System.out.println("【环绕通知-异常】执行失败: " + ex.getMessage());
            
            // 可以选择：
            // 1. 重新抛出异常
            throw ex;
            // 2. 返回默认值
            // return getDefaultValue();
            // 3. 抛出新异常
            // throw new CustomException("包装后的异常", ex);
        } finally {
            // 最终处理
            System.out.println("【环绕通知-最终】清理工作");
        }
    }
}

特点：

  • 最强大的通知类型
  • 可以控制是否执行目标方法
  • 可以修改参数
  • 可以修改返回值
  • 可以捕获并处理异常
  • 必须手动调用 proceed() 执行目标方法
  • 必须返回 Object 类型

通知的执行顺序

单个切面内的执行顺序：

@Aspect
@Component
public class OrderDemoAspect {
    
    @Around("execution(* com.example.service.*.*(..))")
    public Object around(ProceedingJoinPoint joinPoint) throws Throwable {
        System.out.println("1. Around - 前");
        try {
            Object result = joinPoint.proceed();
            System.out.println("5. Around - 返回后");
            return result;
        } catch (Exception ex) {
            System.out.println("5. Around - 异常");
            throw ex;
        } finally {
            System.out.println("6. Around - 最终");
        }
    }
    
    @Before("execution(* com.example.service.*.*(..))")
    public void before() {
        System.out.println("2. Before");
    }
    
    @AfterReturning("execution(* com.example.service.*.*(..))")
    public void afterReturning() {
        System.out.println("4. AfterReturning");
    }
    
    @AfterThrowing("execution(* com.example.service.*.*(..))")
    public void afterThrowing() {
        System.out.println("4. AfterThrowing");
    }
    
    @After("execution(* com.example.service.*.*(..))")
    public void after() {
        System.out.println("5. After");
    }
}

// 正常执行时的输出：
// 1. Around - 前
// 2. Before
// 3. [目标方法执行]
// 4. AfterReturning
// 5. After
// 5. Around - 返回后
// 6. Around - 最终

// 异常时的输出：
// 1. Around - 前
// 2. Before
// 3. [目标方法抛异常]
// 4. AfterThrowing
// 5. After
// 5. Around - 异常
// 6. Around - 最终

Aspect

定义：切点（Pointcut）+ 通知（Advice）的组合，是 AOP 的核心模块化单元。

切面的组成要素：

  1. @Aspect 注解：标识这是一个切面类
  2. 切点定义：使用 @Pointcut 定义可复用的切点
  3. 通知方法：定义在切点上执行的逻辑
  4. 优先级：使用 @Order 控制多个切面的执行顺序

切面的组成要素：

要素说明必需
@Aspect标识切面类✅
@Component注册为 Spring Bean✅
@Pointcut定义可复用的切点❌
@Advice定义通知方法✅
@Order指定优先级❌

Weaving

定义：将切面应用到目标对象并创建代理对象的过程。

源代码 → 编译 → 字节码 → 类加载 → JVM运行
          ↓              ↓         ↓
       编译期织入    类加载期织入  运行期织入
       (AspectJ)    (AspectJ)   (Spring AOP)

AOP Proxy

定义：Spring AOP 通过代理模式实现，代理对象包装目标对象，在方法调用前后执行增强逻辑。

Spring AOP 支持两种代理方式：

特性JDK 动态代理CGLIB 代理
实现方式基于接口基于继承
要求目标类必须实现接口目标类不能是 final
生成的代理类实现目标接口继承目标类
性能创建快，调用稍慢创建慢，调用快
方法限制只能代理接口方法不能代理 final 方法
Spring 默认有接口时使用无接口时使用
强制使用 CGLIB 代理
// 方式1：配置类
@Configuration
@EnableAspectJAutoProxy(proxyTargetClass = true)  // ← 强制使用 CGLIB
public class AopConfig {
}

// 方式2：application.properties
spring.aop.proxy-target-class=true

最佳实践

最佳实践清单

  • ✅ 使用命名切点，提高可读性和复用性
  • ✅ 根据需求选择合适的通知类型
  • ✅ 使用 @Order 控制切面执行顺序
